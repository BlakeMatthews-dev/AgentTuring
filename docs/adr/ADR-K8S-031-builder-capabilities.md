# ADR-K8S-031 — Builder capabilities: Forge's expanded role in the catalog model

**Status:** Proposed
**Date:** 2026-04-09
**Deciders:** Stronghold core team

## Context

Stronghold's catalog model (ADR-K8S-021 for tools, ADR-K8S-022 for
skills, ADR-K8S-027 for agents) provides structured registries for the
three artifact types that agents consume and produce. Each catalog has a
trust-tier system: artifacts enter at a tier reflecting their provenance
and are promoted through review gates toward higher trust levels. The
catalogs answer "what exists and how much should we trust it?" but they
do not answer "how do new artifacts get created?"

Human developers can write tools, skills, and agents by hand — authoring
Python code, Dockerfiles, Helm subcharts, and markdown skill documents,
then committing them to the repository and deploying them through the
normal CI/CD pipeline. This works, but it is slow. In an enterprise with
hundreds of tenants, each with unique integration needs, the velocity of
human-authored artifacts cannot keep up with the demand for new tools,
skills, and agents.

Forge is Stronghold's builder agent. It already exists at trust tier T0,
meaning the platform trusts Forge itself to run with full privileges. But
Forge's outputs are a different matter: code generated by an LLM has not
been reviewed by a human, has not been tested in production, and may
contain bugs, security vulnerabilities, or hallucinated dependencies.
The trust-tier model handles this distinction cleanly — Forge runs at
T0, but everything it produces enters the catalog at skull tier, the
lowest trust level, pending promotion.

What Forge has lacked until now is a structured understanding of what it
can produce and where those outputs land. Under the catalog model, Forge
gains three specific output types, each with a defined home in the
catalog hierarchy and a defined path through the trust-tier promotion
workflow.

## Decision

**Forge produces three kinds of catalog entries — skill artifacts, agent
strategies, and MCP server scaffolds — all entering their respective
catalogs at skull tier and following the standard promotion workflow.**

### Output type 1: Skill artifacts

Forge generates markdown skill documents with YAML frontmatter conforming
to the Skill Catalog schema (ADR-K8S-022). A skill artifact includes:

- A frontmatter block with `name`, `description`, `category`, `tags`,
  `input_schema`, `output_schema`, and `trust_tier: skull`
- A markdown body containing the skill's instructions, examples, and
  constraints
- Optional embedded tool references (names of tools from the Tool
  Catalog that the skill may invoke)

Skill artifacts are deposited into the Skill Catalog via the catalog's
registration API. They are immediately visible in the catalog at skull
tier but cannot be used by agents above skull tier until promoted. An
operator or tenant admin reviews the skill, verifies its correctness,
and promotes it to T3, then T2, and so on through the standard workflow.

The value of Forge-generated skills is velocity: a tenant admin can
describe a business process in natural language, and Forge produces a
skill document that encodes that process as a reusable agent instruction.
The admin reviews and promotes it rather than authoring it from scratch.

### Output type 2: Agent strategies

Forge generates new Python strategy implementations paired with A2A
Agent Cards (ADR-K8S-027). An agent strategy output includes:

- A Python module implementing one of the standard strategy interfaces
  (direct, react, plan_execute, delegate, or custom)
- An A2A Agent Card in JSON format describing the agent's capabilities,
  input/output schemas, and metadata
- A test suite exercising the strategy's core behavior

The strategy module and Agent Card are deposited into the Agent Catalog
at skull tier. The agent is registered in the `agent_registry` table
with `trust_tier = skull` and associated with the requesting tenant.

At skull tier, the agent can only be invoked in a maximally sandboxed
environment: restricted network access, limited token budget, full audit
logging, and no access to production data. This sandbox allows the
tenant to evaluate the agent's behavior before promoting it. Each
promotion step relaxes restrictions: T3 gets limited production data
access, T2 gets standard resource limits, T1 gets elevated privileges,
and T0 is reserved for operator-reviewed agents.

### Output type 3: MCP server scaffolds

Forge generates complete MCP server packages for custom tool backends.
A scaffold output includes:

- A Python FastAPI application implementing the MCP server protocol
  with one or more tool definitions
- A Dockerfile for containerizing the server
- A Helm subchart for deploying the server as a sidecar or standalone
  pod in the Stronghold cluster
- A tool catalog entry (ADR-K8S-021) for each tool the server exposes
- A test suite exercising the tool implementations

MCP server scaffolds are the most complex Forge output. They are
deposited as container image candidates at skull tier: the image is
built but not deployed. An operator reviews the generated code,
runs the test suite, inspects the Dockerfile for security issues, and
promotes the scaffold to T3. At T3, the MCP server is deployed in a
sandboxed namespace with restricted network policies. Further promotion
follows the standard workflow.

The use case is custom integrations: a tenant needs a tool that connects
to their internal API (a proprietary CRM, a legacy database, a
domain-specific service). Rather than waiting for Stronghold to ship a
built-in tool, the tenant asks Forge to generate an MCP server that
wraps their API. The generated server goes through trust-tier promotion,
and once promoted, it is available to the tenant's agents as a
first-class tool.

### Trust-tier promotion workflow

All three output types follow the same promotion path:

```
skull -> T3 -> T2 -> T1 -> T0
```

Each promotion step requires:

1. **Review** — a human (operator or tenant admin, depending on the
   target tier) examines the artifact. For code artifacts (strategies,
   MCP servers), this includes code review. For skill artifacts, this
   includes correctness verification.

2. **Testing** — the artifact must pass its test suite at the current
   tier's sandbox level. Promotion from skull to T3 requires passing
   tests in the skull sandbox. Promotion from T3 to T2 requires passing
   tests with T3 resource limits. And so on.

3. **Approval** — the reviewer explicitly approves the promotion. The
   catalog entry's `trust_tier` is updated, and the change is recorded
   in the audit log.

Promotion is not automatic. The LLM generates; the governance layer
gates. This separation is the core design principle: Forge is fast but
untrusted, humans are slow but trusted, and the trust-tier system
bridges the gap.

### Forge's own trust tier

Forge itself runs at T0 — it is an operator-deployed, human-reviewed
agent that is part of Stronghold's built-in agent set. Its elevated
trust allows it to access the catalog registration APIs, read existing
catalog entries for context, and invoke code-generation LLM models at
the balanced tier. But Forge's trust level does not transfer to its
outputs. A T0 agent producing skull-tier artifacts is not a
contradiction; it is the design. The agent is trusted to generate
candidates; the candidates are not trusted until reviewed.

## Alternatives considered

**A) Auto-promoting Forge outputs to match Forge's own trust tier.**

- Rejected: violates the trust-tier model's core premise. Forge runs
  at T0 because a human deployed and reviewed Forge itself. Forge's
  outputs are LLM-generated code that no human has reviewed. Granting
  T0 trust to unreviewed LLM output would make every Forge-generated
  artifact a potential security vulnerability running with full
  platform privileges.

**B) Humans-only building — no LLM-generated artifacts, all tools,
skills, and agents authored by developers.**

- Rejected: too slow for the velocity of artifact development in a
  large multi-tenant deployment. Each tenant has unique integration
  needs. A developer-only pipeline bottlenecks on human availability
  and creates a backlog that grows faster than developers can clear.
  Forge-generated candidates with human review gates give the
  velocity of LLM generation with the safety of human approval.

**C) Code-only building — Forge generates only Python code, not
markdown skill documents or Helm charts.**

- Rejected: skills are markdown documents, not code. Restricting Forge
  to code generation means skill creation remains a manual process.
  Skills are often the simplest artifacts (natural-language instructions
  with structured frontmatter), and they are precisely the kind of
  artifact that LLMs excel at generating. Excluding them from Forge's
  output types leaves value on the table.

**D) Separate builder agent per artifact type — a tool-forge for MCP
servers, a skill-forge for skills, an agent-forge for strategies.**

- Rejected: fragments the builder capability without clear benefit.
  Forge already has the context to generate all three artifact types
  (it can read existing catalog entries, understand the schema for each
  type, and invoke appropriate LLM models). Splitting into three
  builder agents triples the registration, configuration, and
  monitoring burden without improving output quality. A single builder
  with multiple output types is simpler to operate and reason about.

## Consequences

**Positive:**

- Tenants can request new tools, skills, and agents through natural
  language, dramatically reducing the time from "I need X" to "X is
  available in the catalog."
- Every Forge output enters at skull tier, ensuring that no
  LLM-generated artifact runs in production without human review.
- The promotion workflow is identical for all artifact types, reducing
  the cognitive load on operators who review and promote artifacts.
- Forge's structured output types (with test suites, frontmatter, and
  Helm charts) make review easier than reviewing freeform LLM output.

**Negative:**

- Forge's output quality depends on the LLM's code generation
  capabilities. Poor-quality outputs waste reviewer time. Mitigated by
  Forge's internal quality checks (running the test suite before
  depositing the artifact) and by the skull-tier sandbox that makes
  bad outputs safe to evaluate.
- MCP server scaffolds include Dockerfiles and Helm charts, which are
  security-sensitive artifacts. A Forge-generated Dockerfile could
  reference malicious base images or install unvetted packages. The
  review gate at each promotion step must include Dockerfile inspection.
- The promotion workflow adds latency between "Forge generated it" and
  "it is usable in production." This is by design (the governance
  layer is the quality gate), but tenants may perceive it as slow.

**Trade-offs accepted:**

- We accept that Forge outputs start at skull tier (unusable without
  promotion) in exchange for the safety guarantee that unreviewed
  LLM-generated code never runs at elevated trust levels.
- We accept the review burden on operators and tenant admins in
  exchange for the velocity benefit of LLM-generated artifact
  candidates.
- We accept Forge's dependency on LLM code generation quality in
  exchange for the ability to produce artifacts at a pace that
  human-only development cannot match.
